# **VEFAS: Verifiable Execution Framework for Agents**

*A production-grade multi-platform zkTLS client with strict TDD development*

---

## **1. Vision & Goal**

AI Agents often hallucinate external actions (e.g., claiming "Email sent" without actually sending it).
**VEFAS** eliminates this by making external requests **cryptographically verifiable**:

> At time `T`, with a given request `{method, headers, query, body}`,
> I sent it to resource `https://abc.xyz` over TLS,
> and received the response `{status, payload}`.

The proof is portable and verifiable by anyone — **no MPC, no notary, no trust in gateway**.

---

## **2. High-Level Architecture**

* **AI Agent** → issues command (e.g., send email).
* **VEFAS Gateway** (Multi-platform zkVM support):

    * **SP1 zkVM**: Executes TLS handshake + HTTP exchange with SP1 precompiles
    * **RISC0 zkVM**: Executes TLS handshake + HTTP exchange with RISC0 precompiles
    * Verifies certificate chain using platform-specific crypto implementations
    * Captures request + response
    * Produces platform-specific zkProof
* **Verifier (Node / Blockchain / API)**:

    * Validates zkProof from any supported platform
    * Confirms execution actually happened

## **3. TLS Session Verification Flow (Multi-Platform zkVM)**

### Step 1 — **Handshake Initialization**

* Parse ClientHello + ServerHello.
* Extract cipher suite, key share, extensions.
* **SP1 Precompiles**: `secp256r1` (ECDHE), `sha256/sha384` (transcript hash)
* **RISC0 Precompiles**: `secp256r1` (ECDHE), `sha256/sha384` (transcript hash)

### Step 2 — **Server Authentication**

* Parse X.509 cert chain (ASN.1 DER).
* Verify chain signatures against baked-in roots.
* Match domain (SAN/CN) with target domain.
* **SP1 Precompiles**: `sha256/sha384` (cert digest), `secp256r1`, `ed25519`, `rsa` (cert signatures)
* **RISC0 Precompiles**: `sha256/sha384` (cert digest), `secp256r1`, `ed25519`, `rsa` (cert signatures)

### Step 3 — **Session Key Derivation**

* Compute `ECDHE_shared = client_priv * server_pub`.
* Derive handshake + master secrets via HKDF.
* **SP1 Precompiles**: `secp256r1` scalar multiplication, `sha256` for HKDF extract/expand
* **RISC0 Precompiles**: `secp256r1` scalar multiplication, `sha256` for HKDF extract/expand

### Step 4 — **HTTP Request Commitment**

* Encode full HTTP request.
* Encrypt with session key.
* **SP1 Precompiles**: `aes-gcm` (encrypt request), `sha256` (commit request plaintext)
* **RISC0 Precompiles**: `aes-gcm` (encrypt request), `sha256` (commit request plaintext)

### Step 5 — **Response Capture**

* Decrypt TLS records into plaintext response.
* Parse `{status, headers, body}`.
* **SP1 Precompiles**: `aes-gcm` (decrypt), `sha256` (commit response plaintext)
* **RISC0 Precompiles**: `aes-gcm` (decrypt), `sha256` (commit response plaintext)

### Step 6 — **Proof Generation**

* zkVM commits structured claim:

  ```json
  {
    "domain": "abc.xyz",
    "request_commitment": "...",
    "response_commitment": "...",
    "status_code": 200,
    "tls_version": "1.3",
    "cipher_suite": "TLS_AES_256_GCM_SHA384",
    "certificate_chain_hash": "...",
    "handshake_transcript_hash": "...",
    "timestamp": 1234567890,
    "execution_metadata": {
      "cycles": 1000000,
      "memory_usage": 2048,
      "execution_time_ms": 150,
      "platform": "sp1",
      "proof_time_ms": 75
    }
  }
  ```
* zkProof generated by platform-specific zkVM and returned.

### Step 7 — **Verification**

* Verifier node checks proof using platform-specific verification.
* Confirms: request, response, domain, cert chain, execution metadata.

## **4. Multi-Platform Architecture**

### **4.1. Core Crate Structure**

```
crates/
├── zktls-core/           # Core TLS/HTTP/X.509 logic (platform-agnostic)
├── zktls-crypto/         # Cryptographic trait definitions
├── zktls-crypto-sp1/     # SP1-optimized crypto implementations
├── zktls-crypto-risc0/   # RISC0-optimized crypto implementations
├── zktls-zkvm/           # Platform-agnostic guest logic
├── zktls-sp1/            # SP1-specific host/guest integration
├── zktls-risc0/          # RISC0-specific host/guest integration
└── zktls-gateway/        # Unified API/CLI layer
```

### **4.2. Platform-Specific Precompiles**

| Precompile        | SP1 Usage | RISC0 Usage | Purpose |
| ----------------- | --------- | ----------- | ------- |
| **sha256/sha384** | ✅ | ✅ | Transcript hash, Finished, HKDF, cert digest |
| **aes-gcm**       | ✅ | ✅ | Encrypt request / decrypt response |
| **secp256r1**     | ✅ | ✅ | ECDHE shared secret, ECDSA signatures |
| **ed25519**       | ✅ | ✅ | Optional cert signatures |
| **rsa**           | ✅ | ✅ | Legacy TLS certs (optional) |
| **bigint ops**    | ✅ | ✅ | Modular arithmetic in crypto ops |

**Documentation References:**
- SP1 Precompiles: https://docs.succinct.xyz/docs/sp1/optimizing-programs/precompiles
- RISC0 Precompiles: https://dev.risczero.com/api/zkvm/precompiles

### **4.3. Platform-Specific Implementations**

**SP1 Structure:**
```
zktls-sp1/
├── program/              # Guest program (zkVM entry point)
├── script/               # Host script (proof generation/verification)
└── src/                  # SP1-specific host logic
```

**RISC0 Structure:**
```
zktls-risc0/
├── methods/
│   ├── guest/            # Guest program (zkVM entry point)
│   └── src/              # RISC0-specific host logic
└── src/                  # RISC0-specific host logic
```

## **6. Development Principles**

### **6.1. TLS Knowledge Sources**
Always refer to these authoritative sources for TLS knowledge:
- **RFC 8446 (TLS 1.3)**: https://datatracker.ietf.org/doc/html/rfc8446
- **RFC 8448 (TLS 1.3 Handshake Traces Sample)**: https://datatracker.ietf.org/doc/html/rfc8448
- **TLS1.3 bytes explained**: https://tls13.xargs.org/#open-all

### **6.2. Sequential Thinking (MCP style)**
Always break down TLS verification step-by-step before coding:
1. **Handshake Initialization** → Parse ClientHello/ServerHello
2. **Server Authentication** → Verify certificate chain
3. **Session Key Derivation** → ECDHE + HKDF
4. **HTTP Request Commitment** → Encrypt + commit
5. **Response Capture** → Decrypt + parse
6. **Proof Generation** → Structured zkVM commitment
7. **Verification** → Proof validation

### **6.3. Context7 Rule**
Always pull in RFC 8446 (TLS 1.3), platform-specific documentation, and latest cryptographic best practices before implementation.

### **6.4. Production Quality Standards**

* **No mocks/stubs in production code** - All cryptographic operations must be real
* **Cryptography must be real, correct, and deterministic** - Suitable for zkVM execution
* **Rust best practices**: Idiomatic `Result<T, E>`, zero `unsafe` unless mandatory
* **Platform-specific optimizations**: Leverage zkVM precompiles for performance
* **Comprehensive error handling**: Graceful failure modes with detailed error messages
* **Memory efficiency**: Minimize allocations in zkVM guest programs

## **7. MVP Scope**

### **7.1. Supported Platforms**
* **SP1 zkVM**: Full support with SP1 precompiles
* **RISC0 zkVM**: Full support with RISC0 precompiles

### **7.2. TLS Protocol Support**
* **TLS 1.3 only** (RFC 8446)
* **Cipher suites**: `X25519 + AES-GCM + SHA-256 + ECDSA(P-256)`
* **Key exchange**: ECDHE with X25519 or P-256

### **7.3. Input Requirements**
* **Client ephemeral key** as input (for deterministic proof generation)
* **TLS handshake transcript** (ClientHello/ServerHello)
